# Stage 3 Characters Upgrade — Handoff Document

*Created: 2026-03-22 | For: Claude Code in Antigravity*

---

## Summary

Upgrade Stage 3 (Characters) to use MBTI and Enneagram personality frameworks **under the hood** to generate richer character profiles and deeper downstream data — without exposing any jargon to the user. The visible character card gets a targeted redesign (not a rewrite), and a new hidden data layer powers Stages 4–9.

---

## Design Principles

1. **MBTI + Enneagram are engines, not outputs.** The user never sees type codes. They see results in screenwriting terms.
2. **The card stays clean.** No field bloat. We're rearranging and refining, not adding a wall of new text.
3. **Tags are starting points, not constraints.** The AI pre-selects behavioral tags. The user can override with free text at any time, even if it "breaks" the type.
4. **Hidden layer does the heavy lifting.** A `_deep_profile` object in the JSON stores MBTI/Enneagram data, dialogue fingerprints, relationship dynamics, and scene behavior predictions. Downstream stages read this. The user never manages it.
5. **Ticks are optional.** Not every character needs one. When present, they have a frequency gate.

### Key Changes from Current Schema

1. **`paradox`** moves from `voice_and_behavior` → `psychological_core` (it's a character trait, not a speech pattern)
2. **Three new tag fields** in `voice_and_behavior`: `voice_tag`, `pressure_tag`, `humor_tag` — pre-selected from curated options, user-adjustable
3. **New `arc` object**: `core_drive` (Enneagram-derived, tag-selectable) + `direction` (Growth / Decline / Circular)
4. **`subtlety_guidelines`** replaced by `ticks` object with `enabled` boolean, `description`, and `frequency_gate`
5. **New `_deep_profile`** object: hidden from UI, generated by agent, consumed by downstream stages
6. **`speech_patterns` and `deflection_tactic`** stay as editable text fields — they're already great

### Tag UI Implementation

Tags should render as **pill-style selectors** — the current value shown as a styled pill. On click, a dropdown appears with the curated options. The user can also start typing to enter a custom value not in the list (combobox behavior). This is similar to how other stages use dropdowns but with free-text fallback.

If a user types a custom value that doesn't match any curated option, the tag still works — it just won't map cleanly to MBTI/Enneagram in the hidden profile. The deep profile generation should handle this gracefully (best-effort type inference from custom text).

### Ticks Section

- **Collapsed by default** with a `▶ Ticks` toggle header
- If `ticks.enabled` is `false`, the section shows only "No ticks for this character" with an "Add tick" button
- If `ticks.enabled` is `true`, shows `description` textarea + `frequency_gate` textarea
- The `frequency_gate` should always start with something like "Only surfaces when..." to enforce the gating behavior

### Edit/Approve Behavior (unchanged logic)

Any change to a tag, text field, or tick toggles the Approve button back to active (existing `input` event listener pattern). Approval scrapes the card and saves to JSON (existing `scrapeCharacters()` flow — needs to be updated for new fields).

## Skill File Changes (skill_stage3_characters.md)

### Add to the System Instruction

New section after Rule 4, before output:

```markdown
### Rule 5: Behavioral Engine (Internal — NOT for user display)

After generating each character's visible profile, internally assign:

1. **MBTI Type** — inferred from voice style, decision-making pattern, and social orientation. Use MBTI to determine:
   - Cognitive processing style (how they take in information, make decisions)
   - Communication tendencies (drives the voice_tag and speech_patterns)

2. **Enneagram Type + Wing** — inferred from core_drive, fear, and pressure response. Use Enneagram to determine:
   - Core motivation and deepest fear (drives psychological_core)
   - Stress arrow: what behavior emerges when the character is under maximum pressure
   - Growth arrow: what behavior emerges when the character is growing/healing
   - These map directly to the character's arc across the screenplay

3. **Generate the `_deep_profile`** object using the inferred types:
   - `dialogue_fingerprint`: Concrete writing rules derived from MBTI — preferred sentence length, vocabulary domain, question style, interruption tendency, what topics they avoid
   - `relationship_dynamics`: For each OTHER character in the cast, describe how this character's type interacts with theirs — where they clash, where they align, what triggers conflict
   - `scene_behavior_predictions`: Given this character's Enneagram type, how do they behave in low-stakes vs. high-stakes scenes? What does their stress arrow look like in the Act 2 midpoint? What does their growth arrow look like in the climax?

**CRITICAL:** The `_deep_profile` is for downstream agent consumption only. The user will never see it. Write it in technical, instruction-style language that a drafting agent can follow.

### Rule 6: Ticks Are Optional

Not every character needs a physical tic or behavioral tell. Only assign a tick if the character's psychological profile genuinely produces one — a defense mechanism that protects their Lie. Characters without a meaningful tick should have `ticks.enabled: false`.

When a tick IS assigned:
- The `frequency_gate` must specify exactly when it surfaces (e.g., "Only when she is alone and feels financially trapped")
- Include the existing downstream warning: "WARNING TO DOWNSTREAM AGENTS: This tick must be used a maximum of ONCE per sequence, and only during the scene of absolute highest stress within that sequence."
- The tick MUST evolve or disappear as the character's arc completes (specify when in `frequency_gate`)
```

### Tag Pre-Selection Logic

The agent should:
1. Generate the full psychological core first
2. Internally infer MBTI + Enneagram types from the psychological core
3. Select the closest matching tags for `voice_tag`, `pressure_tag`, `humor_tag`, and `core_drive` from the curated options
4. Generate `speech_patterns` and `deflection_tactic` text fields informed by the type inference but written in concrete, story-specific terms
5. Generate the `_deep_profile` last (it depends on all of the above)

## Re-Generation Logic (New Feature)

### When Character Data Changes After Approval

When the user edits and re-approves Stage 3, the system should NOT auto-cascade to Stage 4. Instead:

1. **Detect what changed** — compare the newly approved characters against the previously saved version. Track which characters changed and which fields.

2. **Present options to the user** — a modal or inline prompt:
   - **"Apply at Stage [N]"** — Skip re-generation of intermediate stages. Go directly to the stage where the user wants the changes to take effect (e.g., Stage 9 for a rewrite). The updated character data is saved and available to that stage.
   - **"Re-generate Stage 4 (surgical)"** — Keep existing beats mostly as-is. Only adjust beats that involve the changed character(s). Pass the diff as context to the agent: "The following character traits changed: [list]. Revise only the beats affected by this change."
   - **"Re-generate Stage 4 (full)"** — Completely regenerate the beat sheet with the new character data. (Existing behavior.)

3. **The default should be "Apply at Stage [N]"** — most users changing character data mid-pipeline want to fix something in a later stage, not redo everything.

### Implementation Notes

- Store a `_last_approved` snapshot of characters in the project JSON when Stage 3 is approved
- On re-approval, diff `_last_approved` against the new data
- The modal should show which characters changed and which fields
- "Surgical" mode passes the diff + existing beats to the Stage 4 agent with a revision prompt (similar to how Stage 3 itself has surgical revision mode)

## Migration / Backward Compatibility

Existing projects have character data in the old schema. The app should handle both:

1. If `_deep_profile` is missing, character data still works — downstream stages just don't get the enriched data
2. If `ticks` is missing, fall back to reading `subtlety_guidelines` (old field)
3. If tag fields are missing, render the text fields only (no tag pills)
4. A "Regenerate" action on Stage 3 for an existing project will produce the new schema and populate all new fields

No data migration script needed — the schema upgrade happens naturally when characters are regenerated.
